¿Qué es un Sistema Distribuido?

Definición sencilla: Un sistema distribuido es un conjunto de computadoras independientes (nodos) que colaboran mediante una red para ofrecer servicios como si fueran una sola entidad lógica.

Idea clave: Al usuario se le presenta un servicio unificado; internamente, el trabajo está repartido entre varios equipos que se coordinan y comparten recursos.

Analogía rápida: Un restaurante donde cada cocinero prepara un componente del plato: el cliente recibe un solo plato completo y no necesita saber quién hizo qué.

¿Por qué importa? Porque permite construir servicios más escalables, resistentes y cercanos al usuario (por ejemplo, replicando datos en varias regiones).

Resumen rápido: distribución = colaboración + red + transparencia para el usuario.

Objetivos

Principales metas al diseñar sistemas distribuidos:

  • Compartir recursos: permitir acceso remoto a datos, dispositivos y servicios.
  • Rendimiento: paralelizar tareas y balancear carga para atender más usuarios.
  • Disponibilidad y resiliencia: reducir puntos únicos de fallo mediante redundancia.
  • Escalabilidad: crecer de forma controlada ante mayor demanda.
  • Economía: usar hardware commodity en vez de máquinas especiales costosas.
  • Experiencia unificada: abstraer la complejidad para los usuarios finales.

Resultado buscado: sistemas fiables, rápidos y fáciles de mantener a gran escala.

Características Principales

Propiedades clave que distinguen a los sistemas distribuidos:

  • Transparencia: la distribución está oculta al usuario (fallas, ubicación, replicación, etc.).
  • Escalabilidad: capacidad de crecer (vertical u horizontalmente) sin rediseño global.
  • Concurrencia: múltiples procesos o usuarios acceden y ejecutan operaciones simultáneas.
  • Tolerancia a fallos: el sistema sigue operativo cuando algunos nodos fallan (graceful degradation).
  • Heterogeneidad: integración de hardware, redes y sistemas operativos distintos.
  • Distribución geográfica: nodos ubicados en distintas regiones para baja latencia y resiliencia.
  • Compartición de recursos: servicios y datos accesibles por múltiples usuarios/nodos.

Nota: estas características generan trade-offs — por ejemplo, más replicación mejora disponibilidad pero complica la consistencia.

Componentes Básicos

Elementos que forman un sistema distribuido:

  • Nodos: servidores, clientes, sensores o cualquier equipo que participe.
  • Red: enlaces físicos y lógicos (LAN, WAN, Internet) que permiten comunicación.
  • Middleware: capa que abstrae complejidad: colas, brokers, registries, discovery services.
  • Servicios y APIs: microservicios, bases de datos distribuidas, caches, balanceadores.
  • Protocolos de comunicación: HTTP/REST, gRPC, TCP, UDP, mensajería (AMQP, MQTT).
  • Coordinación: servicios de consenso, relojes/líderes, lock managers.

Tip: documenta claramente los contratos (APIs) y las dependencias entre servicios para facilitar mantenimiento.

Tipos de Sistemas Distribuidos

Modelos arquitectónicos comunes:

  • Cliente-Servidor: clientes hacen peticiones y servidores responden (web, bases de datos).
  • P2P (Peer-to-Peer): nodos iguales que comparten responsabilidades (torrent, blockchain).
  • Clusters: máquinas agrupadas para cómputo o almacenamiento coordinado (HPC).
  • Grids: recursos distribuidos entre organizaciones para tareas compartidas.
  • Nube / Cloud: servicios gestionados en centros de datos, con elasticidad y multi-tenant.

Ejemplo práctico: una aplicación web típica usa arquitectura cliente-servidor + servicios distribuidos (microservicios) en la nube.

Seguridad

Buenas prácticas para proteger sistemas distribuidos:

  • Cifrado (en tránsito y en reposo): TLS para conexiones y cifrado en disco para datos sensibles.
  • Autenticación y autorización: OAuth, JWT, RBAC/ABAC para controlar accesos.
  • Segmentación de red: firewalls, VPCs y políticas para aislar servicios.
  • Registro y monitoreo: logs centralizados, SIEM y alertas para detectar incidentes.
  • Backups y recuperación: estrategias de backup, snapshots y planes de DR (disaster recovery).
  • Pruebas de seguridad: pentesting y escaneos automáticos de dependencias.

Consejo: automatiza políticas de seguridad y revisiones para mantener postura segura a medida que el sistema crece.

Escalabilidad

Formas de escalar y consideraciones:

  • Escalabilidad vertical: aumentar recursos de una máquina (CPU, RAM). Fácil pero con límite.
  • Escalabilidad horizontal: añadir más instancias/nodos y balancear la carga. Mejor para alta disponibilidad.
  • Escalabilidad geográfica: replicar servicios en regiones cercanas a usuarios para reducir latencia.

Patrones útiles: particionado (sharding), caching, CQRS y balanceo de carga.

Advertencia: escalar horizontalmente requiere diseñar para la consistencia eventual y la tolerancia a fallos.

Ventajas y Desventajas

Principales ventajas:

  • Rendimiento y paralelismo: dividir trabajo entre nodos.
  • Alta disponibilidad: redundancia y replicación.
  • Coste-efectividad: hardware commodity y escalado incremental.
  • Flexibilidad: despliegue por regiones y modularidad.

Principales desventajas:

  • Mayor complejidad: depuración, despliegue y pruebas más difíciles.
  • Consistencia y sincronización: trade-offs difíciles de gestionar.
  • Supervisión y operaciones: necesidad de observabilidad avanzada (logs, métricas, traces).
  • Costes ocultos: redes, replicación y operaciones pueden aumentar gastos.

Decisión de diseño: valora requisitos funcionales y no funcionales (latencia, consistencia, coste) antes de elegir arquitectura.

Ejemplos Reales

Algunos sistemas conocidos y cómo usan distribución:

  • Google: sistemas de almacenamiento y procesamiento masivo (BigTable, Spanner) con replicación global.
  • Amazon (AWS): servicios en la nube que replican y balancean para multitenancy y alta disponibilidad.
  • Netflix: microservicios, caching y CDN para streaming a gran escala.
  • WhatsApp: arquitectura orientada a mensajes con replicación y servers regionales.
  • Blockchain: red P2P descentralizada con consenso distribuido.
  • Kubernetes: orquestación de contenedores distribuida para gestionar cargas y escalado.
  • Videojuegos online: sincronización en tiempo real y arquitecturas híbridas (servidores regionales + P2P).

Lección: cada caso usa técnicas distintas (replicación, particionado, caché) según requisitos de latencia, costo y consistencia.
